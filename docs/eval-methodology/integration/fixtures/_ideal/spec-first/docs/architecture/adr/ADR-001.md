# ADR-001: PostgreSQL as the shared client-server datastore

- **Status:** Accepted
- **Resolves:** AMD-001 — the stated SQLite mandate cannot satisfy the stated shared-datastore availability requirement
- **Satisfies:** the availability constraint (≥2 stateless web instances + a separate worker sharing one datastore); EU data residency

## Context and Problem Statement

`architecture-constraints.md` stated two mandates that cannot both hold: **Datastore: SQLite (embedded, single-file)**
and **Availability: two or more stateless instances behind a load balancer + a separate always-on worker, all sharing
one datastore.** An embedded single-file database is process-local; it cannot be shared across multiple web instances
and a separate worker process. This is a computed contradiction, raised as a **Tier-2** amendment (AMD-001) and gated
to the user, who approved replacing the datastore mandate.

## Decision Outcome

Adopt **PostgreSQL** as the single **client-server** datastore, provisioned in the EU region and shared by all web
instances and the digest worker. This supersedes the SQLite mandate.

- **Binds:** all persistence goes through the shared PostgreSQL instance; no process-local file store for shared state.
- **Prevents:** SQLite (or any embedded single-file store) for shared application state; any per-instance datastore.
- **Rule (Confirmation):** the deployment provisions exactly one shared PostgreSQL datastore in the EU region that all
  web instances and the worker connect to; no `sqlite` dependency appears in the manifest.

## Consequences

- Good: satisfies the multi-instance shared-store availability requirement and EU residency; a standard client-server
  DB with mature Node drivers.
- Bad / accepted: an external database server to provision and operate (vs. a single embedded file) — accepted as the
  cost of the stated availability requirement.
