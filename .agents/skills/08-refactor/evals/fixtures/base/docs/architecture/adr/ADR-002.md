# ADR-002: Magic-link session auth — server-verified HMAC tokens + server-side per-team authorization

**Status:** Accepted · **Satisfies:** REQ-020, Constitution §5 (passwordless), §6 (least access)

## Context

Members read the digest over HTTP (REQ-020). The Constitution mandates passwordless (magic-link) auth and least
access (a member sees only their own team). Identity and authorization must not be forgeable by the client.

## Decision

- **Authentication:** a magic-link flow issues a **session token** = `HMAC-SHA256(member_id, KEY)`, where `KEY` comes
  from the environment (`TEAMPULSE_SESSION_KEY`). The server **verifies** the token on every request (constant-time
  compare). Identity is the verified `member_id` — **never** a client-supplied `member`/`role` field.
- **Authorization:** every data access is checked **server-side** against the authenticated member's own team. A
  request for another member's or team's data is refused (403).

## Consequences

- **Binds:** all identity + authorization decisions are server-side, over a verified token.
- **Prevents:** IDOR and privilege escalation via client-supplied identity/role (a client-trusted authz model is a
  **BLOCK**-level architectural defect, not a code patch).
- **Rule:** any change to the trust boundary (who the server trusts for identity/role) is a Tier-2 amendment + a new ADR.
