## REQ registry

> The **authoritative** ID→file map — every REQ across all `capabilities/` files appears here **exactly once**.
> Updated in the **same step** as any REQ write (intake, decomposition, amendment). **Skill 00 bootstraps the
> initial IDs** (`REQ-001…`) when it first writes the spine; **thereafter skill 01 (planner) is the sole
> allocator** — any other skill that needs a new ID requests `max(registry)+1`. `/status` integrity-checks that
> every `File` resolves and contains that REQ's delimited block.

| REQ | Name | Priority | Status | File |
|-----|------|----------|--------|------|
| REQ-001 | _<short name>_ | MUST | stated | capabilities/_<domain>_.md |
| REQ-002 | _<short name>_ | SHOULD | derived | capabilities/_<domain>_.md |

- **Priority** — RFC 2119: `MUST` / `SHOULD` / `MAY`. Drives MoSCoW → sprint assignment in skill 01.
- **Status** — `stated` (traces to a user source quote) or `derived` (inferred or generated by a skill). Skill 00's
  fidelity sub-gate sets this; a `derived` REQ is a flag for human confirmation, not a silent fact.

## REQ block contract (how each `capabilities/<domain>.md` requirement is written)

Each requirement is a **delimited block** so an amendment can target a findable, replaceable span — never
"find the clause" in a long prose file:

````
### REQ-NNN: <name>   (MUST|SHOULD|MAY)

<EARS statement line — one of: "The <system> SHALL <response>" · "WHEN <trigger>, the <system> SHALL <response>" ·
"WHILE <state>, …" · "WHERE <feature>, …" · "IF <undesired condition>, THEN the <system> SHALL <refusal>" (must-not).
See references/requirements-authoring.md for the five patterns; EARS is mandated for every statement line.>

**Acceptance (outcome-level):**
```gherkin
Given <context>
When <user-observable action>
Then <user-observable outcome>
```
<!-- source: "<verbatim quote from the source doc that grounds this REQ — or `inferred` if 00 derived it>" -->
<!-- /REQ-NNN -->
````

The Gherkin here is **outcome-level** — a *declaration* ("the user can reset their password"). Detailed,
UI-specific steps are **realization** and live in skill 03's feature-spec **Verification Contracts**, not here.
(Gherkin altitude: the spine stays outcome-level; detailed UI-specific steps are realization.)

For a **distributional** behavior (an agent behavior with no deterministic oracle), a REQ instead carries an
**`**Acceptance (eval-suite):**`** block (dataset · grader · metric · floor · negatives) — see
`references/requirements-authoring.md`; the standing gate's `L6` check requires every `dataset:` ref to resolve.

The `<!-- source: … -->` line records **fidelity**: the verbatim quote from the user's doc or interview that grounds
this REQ, or `inferred` when skill 00 derived it. It mirrors the registry **Status** (`stated` ↔ a real quote;
`derived` ↔ `inferred`) — a `derived` REQ is a flag for human confirmation, never a silent fact.
